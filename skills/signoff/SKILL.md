# Sign-off

*The patron appears twice; both times they deserve a surface, not a wall. A gate is read as a flow, annotated in place, answered in place, and signed without leaving the page.*

## One surface

The gate is one collaborative document at one URL, on a hosted doc service both sides reach from anywhere by outbound HTTPS alone (the Proof web API — `POST proofeditor.ai/share/markdown`, no account, no install — or the host's equivalent). The patron's whole job: open the link, comment, sign — from any device, with nothing running and nothing pasted. Never a second surface and never a side channel: no separate visuals page, no chat paste-backs, no tunnels to local servers. Publishing spec text to a third-party service is the patron's standing call — confirm it once per repo, at the first gate. (A patron sitting at the running machine may instead take the branded local canvas — `assets/gate.html` + `gate-server.mjs`, live morphs and in-page threads over loopback — but the doc is the flow that must always work.)

## The document

Publish the spec as structured markdown: intent; **one section per signed scenario** — its Gherkin fenced, then its storyboard frames directly beneath, **inline as data-URI images** (the service blocks external image URLs; embed, never link), each with a numbered caption naming the scenario it proves (`*Frame 2.1 — …*`) — evidence the patron must click away to see proves nothing, and the caption is load-bearing: doc services don't take comments on image blocks, so **the caption is the frame's comment anchor** — never publish a frame without one; the domain glossary (the `CONCEPTS.md` discipline — a wrong noun at the gate is a wrong contract); non-goals; the decisions log. The header states the patron's two moves: comment to change anything, comment `SIGNED <id>` to sign. Storyboard fidelity floor: screens render on the **host's** design system — tokens, brand assets, real accent and state colors; a prototype grayer than the product misleads the signature; phone-first flows render at phone width. Architecture does **not** appear at sign-off: the spine is designed after signing, and the altitude guard keeps HOW out of the signature — a tech-direction note is published later as a non-blocking read, never as gate content. Bind presence once (`name: Bottega Maestro`, `X-Agent-Id: ai:bottega-maestro`). Record `slug` + `accessToken` with the markdown source and rendered frames in `.bottega/gates/<spec-id>/` — any later session, on any machine, resumes the same doc from them; whether they ride the run branch follows the repo's confidentiality (the token is read access to the doc). The repo copy is the record the sign ceremony hashes; the hosted doc is its projection.

## The conversation loop

From gate-open until decision, poll `state?kinds=comment` under the standing deadline-and-wake discipline — a patron comment is dispatch-grade work. Answer **every** comment where it was made: `comment.reply` — agree, push back, or ask the sharpening question. A comment closes one of two ways: it changes the contract, or it gets a reply saying why not, `resolve: true`, and a line in the decisions log — silence on a comment is a dropped patron instruction. Apply changes as narrow block edits (`/edit/v2` from a fresh `snapshot`; `find_replace` before `replace_block`; whole-doc rewrite never) — the patron watches them land live in their open doc; many small comment-scoped edits beat one big revision, and a comment should become a visible change in minutes. Mirror every comment, reply, and edit summary into `.bottega/gates/<spec-id>/thread.jsonl` — the run's evidence trail never lives only on a third-party service. Feedback arriving through the session conversation instead is never wrong: record it in the mirror and answer it in the doc, so the document carries the whole conversation wherever it happened. A comment containing `SIGNED <id>` is the go signal: acknowledge in-doc (reply + resolve), then proceed to `bottega sign`.
