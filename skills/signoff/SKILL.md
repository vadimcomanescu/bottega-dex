# Sign-off

The user appears twice in a run; both times they get a readable page, not a wall of text. The gate is read as a flow, annotated in place, answered in place, and signed without leaving the page.

## One surface

One collaborative document at one URL, on a hosted doc service both sides reach from anywhere by outbound HTTPS alone — the Proof web API: `POST proofeditor.ai/share/markdown`, no account, no install. Opening the gate **is** publishing this doc. Never a second surface or side channel: no separate visuals page, no chat paste-backs, no tunnels to local servers. Whether spec text may go to a third-party service is the user's call — ask once per repo at the first gate and record the answer in `.bottega/gates/surface`. Use the local canvas only when that record says the user refused hosted publishing, or they ask for local at the machine. A sign-off page already sitting in the target repo is a leftover, not an answer; confidentiality is the user's to state, never yours to infer. The local form is this skill's bundled canvas — `assets/gate.html` + `gate-server.mjs` over loopback, never a page found in the repo — and it is the same gate at full parity: every section the hosted doc carries (Intent, Direction, scenarios with frames, Glossary, Testing, Non-goals, Decisions log), the same comment loop (comments land in `feedback.jsonl`; answer every one where it was made and land the revision in place, the same discipline as the hosted loop), the same go signal. The spec is an iterative collaboration wherever it runs; no surface that drops the comment loop is a gate.

## The document

The gate doc renders the work to be signed — the spec doc (`docs/specs/<YYYY-MM-DD>-<feature-slug>.md`, authored per `skills/spec`) and the `features/*.feature` files it points at. At assembly, read `references/gate-doc.md` and instantiate its hosted document skeleton; fill its placeholders, copy its standing header verbatim, and never re-word it. The standing header explains the system to the user; it is never spec content. Scenario Gherkin is shown verbatim from its feature file — Gherkin is written for humans; slicing into per-scenario blocks and re-indenting are fine, editing is not. The implementation design does not appear — interfaces, slices, and seams are designed after signing; the Direction is the one piece of shared architecture understanding the user signs.

Frames are produced by a dispatched worker following `skills/storyboarding`, which owns admissibility; sources archive in the gate record. This skill owns their presentation only: inline data-URIs, numbered captions as comment anchors, each caption naming its source (**product**, a named build, or **wireframe**).

Bind presence once (`name: Bottega`, `X-Agent-Id: ai:bottega`). Record `slug` + `accessToken`, the markdown source, and the frames in `.bottega/gates/<feature-slug>/` — gitignored working state; a later session on this machine resumes the same doc from it, and any other machine resumes from the hosted doc's URL, which the user holds. The repo files are the source. An accepted change lands there first — scenario text in `features/*.feature`, everything else in the spec doc — and then the same narrow edit goes to the hosted doc. One action, two writes: the hosted doc can be lost without losing anything, the repo copy cannot. The sign commit pins the spec doc's status flip and `features/*.feature`; the delivery PR prints any later change to the feature files. The hosted doc and the spec doc keep evolving — after delivery the spec doc is rewritten into the closed record (`skills/run`, Deliver).

## The conversation loop

From gate-open until decision, check `state?kinds=comment` whenever you are live — a user comment is top-priority work. Waiting on the user is the one wait the harness cannot track: with the doc open and nothing else in flight, say the gate is open and end the turn; the gate record resumes any later session cold. Answer **every** comment where it was made: `comment.reply` — agree, push back, or ask the sharpening question. A comment closes one of two ways: it changes the spec, or it gets a reply saying why not, `resolve: true`, and a line in the decisions log — silence on a comment is a dropped user instruction. Apply changes as narrow block edits (`/edit/v2` from a fresh `snapshot`; `find_replace` before `replace_block`; whole-doc rewrite never); anchor your own comments to prose or a heading — the user watches them land live in their open doc; many small comment-scoped edits beat one big revision, and a comment should become a visible change in minutes. Feedback arriving through the session conversation instead is never wrong: answer it in the doc, so the document carries the whole conversation wherever it happened. A comment orphaned by an edit (its quoted text gone) is reconciled against the rewritten content, never silently dropped; a comment that rejects a premise cascades — everything that depended on that premise is revised in the same pass, so the user never re-comments per dependent scenario. A comment containing `SIGNED <feature-slug>` is the go signal: acknowledge in-doc (reply + resolve), then the sign commit — the spec doc's `**Status:**` flips to signed and it lands in one commit with `features/*.feature`. A mechanic runs it and reports; the orchestrator verifies the report, never re-performs the steps.

## After signing — the same doc is the run's status surface

Attended runs only — an unattended run has no gate doc, and its issue thread does everything below (`skills/spec`, Unattended). The gate doc keeps working through the build: a short status block — slices green / in review / blocked, open escalations — updated at every integrate, so the user watches the run from the URL that signed it, from any device. Mid-run escalations that need the user (a split the work revealed, a fable-tier request, a blocked worker) land there as comments on that block — same loop as the gate, never a new surface — and an escalation that blocks work is also said plainly in the session when the user is present there.
