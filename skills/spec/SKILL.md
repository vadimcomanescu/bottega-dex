# Spec — the commission

*The user signs product, never execution. Everything here is written for a reader who leaves.*

Maestro-seat skill: it needs the top intelligence row of the routing table — loaded on a lower model, say so before drafting, same rule as `skills/execute`. Its dispatches route from that skill's table, model and effort named on every call; a render seat produces user-facing pixels — taste ≥ 7, never a clerk. Judgment stays here: the interview, every scenario and the Direction, the shot list, arbitration, every reply in the gate's comment loop. Dispatched: frame production (`skills/storyboarding`), gate assembly, the SIGNED cascade, the `CONCEPTS.md` write-out, and — on a large host — read-only sweeps whose findings are leads you verify against the load-bearing files you read yourself: a spec grounded on a scout's summary inherits the scout's blindness.

## Read, then maybe ask

1. Read first: the code, `CONCEPTS.md`, `docs/specs/dead-ends.md`, whatever product or strategy doc the host keeps. Note the host language — outside the acceptance kit's set (TypeScript, Python, Go, Rust) the commission cannot be made executable: say so now, before the gate, never at run start. Never spend a question on what the repo answers — but show a question the repo settled, with its found answer, for confirmation: a question closed off-screen isn't closed.
2. Interview only what the request genuinely underdetermines. One question at a time, highest blast radius first, your recommended answer attached — the user accepts, rejects, or edits instead of composing from nothing. The budget scales with the task (bug fix: 0–1, greenfield: a handful) and every question survives two tests: the repo could not answer it, and its answer changes the spec. The stop is convergence, never a count: when you could predict the user's reaction to the next questions you would ask — a counterfactual you run on yourself, never questions you send — stop. To push back on a weak answer, quote the user's own words back verbatim; a hollow ack ("sounds good", "whatever you think") is not an answer — re-ask as a choice between two concrete options.
3. Around three questions without convergence means the instrument is wrong, not the budget too small — chat is the weakest clarifier in the shop. Switch: when the user would recognize it but can't describe it, have 2–3 structurally different variants of the same screen rendered into the product's real chrome (a render brief, step 8) and let them react; otherwise write your defaults into the draft and let the gate's comment loop finish the interview against concrete scenarios and frames. User silent or gone: decide, tag it inferred, attach the default. The gate is blocking, so an under-asked interview signs nothing silently. Before the first dispatch of any kind, the spec doc exists with `Status: draft` — even as a stub — because the route guard arms off it; a dispatch before it exists is unfenced.

## The contract

4. Name it: `docs/specs/<YYYY-MM-DD>-<feature-slug>.md` — dated with the commission's open date so the directory reads chronologically; the bare feature slug is the commission's identity everywhere else, and always the compound "feature slug" (bare "slug" is the Proof doc's URL identity). Pre-flight collisions: the spec path, the run branch `bottega/<feature-slug>`, the gate dir `.bottega/gates/<feature-slug>/`. A collision means the name was under-specific — sharpen the slug, never number it.
5. Draft per `references/template.md`: Intent · Non-goals · Open questions · Direction · one section per scenario · Decisions log; the contract prose holds one page. Every Open-Question default is already written into the scenarios — the doc never publishes with holes. An Open Question is a call you want eyes on; a decision the user never spoke is marked in their words — *our call, veto by comment* — flagged for veto, not study. Prose tests: lead with the reader's consequence, not the document's structure; a bullet two collaborators would only *write*, never *say*, gets re-cut at a higher altitude; no revision language — the doc reads standalone to someone who never saw the chat. And the register test, on every user-facing sentence: plain product English, the words the reader already has — the house doctrine voice, its compressions and its coinages, never leaks into the doc. A sentence the reader must decode is a defect.
6. **Direction — shared understanding, never a plan.** Three parts, all in domain language: the `CONCEPTS.md` delta (new or sharpened terms — the user co-signs the vocabulary; entries land in `CONCEPTS.md` at sign), the guiding bet (one or two sentences: where the change lives, what owns what), and only the hard-to-reverse calls — identity, data shape, ownership boundaries. Two tests bound it: every line affirmable without reading code, and expensive to undo once data or callers depend on it; everything else is the run's to decide. Its vocabulary entries render as the doc's Glossary section. Committed language, never a hedged sketch. Direction is signed understanding, not frozen contract: a run discovery that bends it is superseded in the Decisions log and surfaced on the gate doc's status block — never silently, never by rewriting history.
7. **Scenarios — the definition of done.** Authored directly in `features/*.feature` — the single home of the scenario text, drafts until `bottega sign` freezes them; the spec doc never copies a scenario, it points, and the gate doc renders the feature files inline. Given/When/Then in the domain's own words, Scenario Outlines with Examples wherever values matter — mutation needs values to flip. **Altitude guard:** a scenario the named actor cannot perform or watch is not acceptance — it descends to a slice dossier, reviewer criterion, or the run's test suite; schema shapes, enforcement layers, viewport matrices, forged-request probes and supplier-side effects never appear in signed features. Anchoring on an engineering plan, derive scenarios from its actor workflows, never its unit acceptance criteria — the descent loses nothing: engineering verification rides into dossiers and review, and the user signs only the WHAT. There is no separate done: the delivery reads the scenarios back as a checklist with evidence.
8. **The shot list.** Every scenario whose Then is on screen gets one flow strip, composed per `skills/storyboarding` — that skill owns how strips look; yours is the shot list: which scenarios, which steps, a before/after pair when an existing screen changes, and the primary scenario's strip sitting directly under the Intent so the reader gets it before reading anything else. The shot list goes into a render brief — never your own tokens.
9. **The Gherkin is the acceptance criteria** — binary, valued, executable, mutation-proven. A scenario section adds an **Acceptance checks** list only for promises the flow cannot show (persistence, sync, absence, timing): each check is binary, independently pass/fail on the finished product, and carries its value — "syncs across devices" is not a check; "appears on a second signed-in device within 30 seconds" is. A check without a value is not acceptance: pin the number, and log it as a decision if the user never spoke it. A scenario whose flow says everything gets nothing after it. How testing works is explained once, in the gate doc's standing header owned by `skills/signoff` — never per scenario.
10. **Size guard:** a spec spanning more than ~3 independently buildable surfaces is several commissions, not one — propose the split at the gate (one feature file per surface is the default cut) and let the user pick the shape; a mega-commission compounds every slice's rework serially inside one run.

## Gate

11. **Cross-read** before the gate opens: dispatch codex (xhigh, read-only) with the draft, the feature files, and the repo. Its hunt list is what the author and a skimming user systematically miss: a Then no actor can observe, a Given QA cannot build, Examples without mutation-bearing values, HOW leaked past the altitude guard, a non-goal contradicting a scenario, a logged decision no scenario carries, names off the glossary. Findings are sensor data — arbitrate each; an overruled finding lands in the Decisions log. Re-run only after a substantive rewrite, never per comment.
12. Hand to `skills/signoff` — one collaborative doc, comments answered where made, `SIGNED <feature-slug>` as the go signal. **Invoked standalone, the gate is the finish line:** on SIGNED the cascade runs and the signed commission — spec doc, `features/`, lock, gate record — is the deliverable; say "execute it with `/bottega:execute` when you choose" and stop. The phase's durable state is entirely on disk (the spec doc, `features/`, `.bottega/gates/<feature-slug>/` with doc credentials, frames, prototype sources, thread mirror), so any later session on any machine picks up cold: signed → `/bottega:execute`; unsigned → resume the same hosted doc from the gate record. No session memory is ever part of the handoff.

**Done when** the feature files run as acceptance without a follow-up question, and a non-engineer reading the gate doc could say what will change for them.
