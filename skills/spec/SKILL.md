# Spec — the commission

The user signs product, never execution. Everything here is written for a reader who leaves.

This is a maestro-seat skill: it needs the routing table's top intelligence row. Loaded on a lower model, say so before drafting (same rule as `skills/execute`). Dispatches route from that skill's table with model and effort named on every call; a render seat produces user-facing pixels and needs taste ≥ 7, never a mechanic. Judgment stays in your turns: the interview, every scenario, the Direction, the shot list, arbitration, every reply in the gate's comment loop. Dispatched: frame production (`skills/storyboarding`), gate assembly, the SIGNED cascade, the `CONCEPTS.md` write-out, and on a large host, read-only sweeps. A sweep's findings are leads you verify against the load-bearing files you read yourself; a spec grounded on a scout's summary inherits the scout's blindness.

## Read, then maybe ask

1. Read first: the code, `CONCEPTS.md`, `docs/specs/dead-ends.md`, whatever product or strategy doc the host keeps. Note the host language: outside the acceptance kit's set (TypeScript, Python, Go, Rust) the commission cannot be made executable; say so now, before the gate, never at run start. Never spend a question on what the repo answers, but do show a question the repo settled, with its found answer, for confirmation: a question closed off-screen isn't closed.
2. Interview only what the request genuinely underdetermines. One question at a time, highest blast radius first, your recommended answer attached so the user accepts, rejects, or edits instead of composing from nothing. The budget scales with the task (bug fix: 0 or 1; greenfield: a handful), and every question survives two tests: the repo could not answer it, and its answer changes the spec. Stop at convergence, never at a count: when you can predict the user's reaction to the next questions you would ask, stop. One probe before declaring convergence: ask what would make them send the delivery back with every scenario green. The answer is a missing scenario, a missing valued check, or the convergence itself. A hollow ack ("sounds good", "whatever you think") is not an answer: re-ask as a choice between two concrete options, and push back on a weak answer by quoting the user's own words back.
3. Three questions without convergence means the instrument is wrong, not the budget too small; chat is the weakest clarifier in the shop. A question born visual (layout, hierarchy, which affordance is primary) never spends chat at all: it goes straight to variants. The switch: when the user would recognize it but can't describe it, have 2 or 3 structurally different variants of the same screen rendered into the product's real chrome (a render brief, step 8) and let them react. Otherwise write your defaults into the draft and let the gate's comment loop finish the interview against concrete scenarios and frames. User silent or gone: decide, tag it inferred, attach the default. The gate is blocking, so an under-asked interview signs nothing silently.

## The contract

4. Name it `docs/specs/<YYYY-MM-DD>-<feature-slug>.md`, dated with the commission's open date so the directory reads chronologically. The bare feature slug is the commission's identity everywhere else; always say "feature slug" (bare "slug" is the Proof doc's URL identity). Pre-flight collisions: the spec path, the run branch `bottega/<feature-slug>`, the evidence branch `bottega/evidence-<feature-slug>`, the gate dir `.bottega/gates/<feature-slug>/`. A collision means the name was under-specific: sharpen the slug, never number it.
5. Draft per `references/template.md`: Intent, Non-goals, Open questions, Direction, one section per scenario, Decisions log. Contract prose holds one page. Every Open-Question default is already written into the scenarios; the doc never publishes with holes. An Open Question is a call you want eyes on, carried with its default and the concrete alternative so overruling costs one word. A decision the user never spoke is written in their words and lands under the log's **Our calls — veto by comment**, flagged for veto, not study. Every user-facing sentence is plain product English — the words the reader already has, no house coinages, nothing they must decode; the doc reads standalone to someone who never saw the chat.
6. **Direction: shared understanding, never a plan.** Three parts, all in domain language: the `CONCEPTS.md` delta (new or sharpened terms the user co-signs; entries land in `CONCEPTS.md` at sign), the guiding bet (one or two sentences: where the change lives, what owns what), and only the hard-to-reverse calls (identity, data shape, ownership boundaries). Two tests bound it: every line affirmable without reading code, and expensive to undo once data or callers depend on it. Everything else is the run's to decide. Its vocabulary entries render as the doc's Glossary section. Committed language, never a hedged sketch. Direction is shared understanding, not a locked plan: a run discovery that bends it is superseded in the Decisions log and surfaced on the gate doc's status block, never silently.
7. **Scenarios: the definition of done.** Authored directly in `features/*.feature`, the single home of the scenario text; drafts until the user approves. The sign commit lands them, and the delivery PR prints any later change to them beside the code that shipped. The spec doc never copies a scenario, it points; the gate doc shows the feature files verbatim — Gherkin is written for humans, and a well-written scenario needs no translation. Steps in the domain's own words, addressed to the actor in second person (`When you save the search "red sneakers"`). Scenario Outlines with Examples wherever values matter; mutation needs values to flip. **Altitude guard:** a scenario the named actor cannot perform or watch is not acceptance; it descends to a slice dossier, reviewer criterion, or the run's test suite. Schema shapes, enforcement layers, viewport matrices, forged-request probes, and supplier-side effects never appear in signed features. Anchoring on an engineering plan, derive scenarios from its actor workflows, never its unit acceptance criteria; engineering verification rides into dossiers and review, and the user signs only the WHAT. There is no separate done: the delivery reads the scenarios back as a checklist with evidence.
8. **The shot list.** Every scenario whose Then is on screen gets one flow strip, composed per `skills/storyboarding` (that skill owns how strips look). Yours is the shot list: which scenarios, which steps, a before/after pair when an existing screen changes, and the primary scenario's strip directly under the Intent so the reader gets it before reading anything else. The shot list goes into a render brief, never your own tokens. The shot list is also the proof list: every listed flow comes back at delivery as a recording of QA driving the shipped product. A flow whose surface has no capture channel is flagged in the doc before sign, never discovered at delivery.
9. **The Gherkin is the acceptance criteria:** binary, valued, executable, mutation-proven. A scenario section adds an **Acceptance checks** list only for promises the flow cannot show (persistence, sync, absence, timing). Each check is binary, independently pass/fail on the finished product, and carries its value: "syncs across devices" is not a check; "appears on a second signed-in device within 30 seconds" is. A check without a value is not acceptance: pin the number, and log it as a decision if the user never spoke it. A scenario whose flow says everything gets nothing after it. How testing works is explained once, in the gate doc's standing header owned by `skills/signoff`, never per scenario.
10. **Testing.** Each scenario section carries one line saying how it gets verified, in the scenario's own plain terms: what gets opened, what gets done, what must be visible, and the evidence that comes back (screenshot, recording, output). Concrete or rejected — "will be tested" says nothing; "open /search signed in, save 'red sneakers', screenshot Saved searches showing it" is a plan. Anything worth probing beyond the scenarios gets the same one-line treatment. The QA seat inherits these lines as its drive plan. Unit testing is the builders' discipline and never appears in the contract.
11. **Size guard:** a spec spanning more than ~3 independently buildable surfaces is several commissions, not one. Propose the split at the gate (one feature file per surface is the default cut) and let the user pick the shape; a mega-commission compounds every slice's rework serially inside one run.

## Gate

12. **Cross-read** before the gate opens: dispatch codex (xhigh, read-only) with the draft, the feature files, and the repo; headless `codex exec` per `codex-dispatch.md` in `skills/execute`, never a codex plugin or wrapper agent the host happens to carry. Its hunt list is what the author and a skimming user systematically miss: a Then no actor can observe, a Given QA cannot build, Examples without mutation-bearing values, HOW leaked past the altitude guard, a non-goal contradicting a scenario, a logged decision no scenario carries, names off the glossary, an *our call* one question would have retired, an actor a Given names whom the interview never touched, a zero, a many, or a failure no scenario visits, a testing line with no how. Findings are sensor data: arbitrate each; an overruled finding lands in the Decisions log. Re-run only after a substantive rewrite, never per comment.
13. Hand to `skills/signoff`: one collaborative doc, comments answered where made, `SIGNED <feature-slug>` as the go signal. **Invoked standalone, the gate is the finish line:** on SIGNED the cascade runs and the signed commission (spec doc marked signed, `features/`, gate record) is the deliverable. Say "execute it with `/bottega:execute` when you choose" and stop. The phase's durable state is entirely on disk (the spec doc, `features/`, `.bottega/gates/<feature-slug>/` with doc credentials, frames, prototype sources, thread mirror), so any later session on any machine picks up cold: signed → `/bottega:execute`; unsigned → resume the same hosted doc from the gate record. No session memory is ever part of the handoff.

**Done when** the feature files run as acceptance without a follow-up question, and a non-engineer reading the gate doc could say what will change for them.
