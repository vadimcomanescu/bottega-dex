Intent

Anyone shopping can save a search and find it again under Saved searches — across devices, across sessions. Today a good query is retyped from memory or lost; regulars rebuild the same search weekly.

Direction — the one bet you co-sign

A saved search belongs to your account, not to a device. The list lives with your profile and follows you; search itself — how results are ranked and shown — is untouched by this work.

Glossary

  • saved search — a query stored on your account, reopenable from any signed-in device.
  • Saved searches — the list of them, newest first, reachable from your profile.

Acceptance checks — verified by the tester on the finished product

  • A search saved on your phone appears on a second signed-in device within 30 seconds.

Testing strategy — how these promises get attacked

Automated
Both scenarios above run as generated acceptance tests on every build of the run.
Load-bearing values
"red sneakers", "size 44 sandals", and the 30-second sync window — each gets flipped in CI and the suite must fail, proof the tests read this exact contract.
Edges to probe
Saving the same query twice keeps one entry, not two. An empty search can't be saved — the control stays disabled. A list of 300 saved searches still opens in under a second.
Manual (QA)
Drive both flows at phone width. Try to save while signed out and expect the sign-in prompt. Watch the second-device sync live, recorded.
Not tested
Search result relevance itself — owned by the existing search suite, unchanged by this commission.

Not in this commission

  • No alerts or notifications when a saved search has new results — a later commission if wanted.
  • No sharing a saved search with another account.

Open question — overrule by one word

List order — default: newest first (already written into the scenarios) · or: alphabetical. Comment either word; signing accepts the default.

Decisions log

You said

  • Saved searches are per-account, never per-device — the whole point is finding them on the laptop.

Our calls — veto by comment

  • The list caps at 300, oldest drops first — keeps it scannable; nobody rebuilds search #301 from memory.
  • Saving an identical query twice keeps one entry and bumps it to the top — duplicates would bury the list.
